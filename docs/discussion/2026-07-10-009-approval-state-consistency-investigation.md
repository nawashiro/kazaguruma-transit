# 009 承認状態不整合の調査と修正計画

## 結論

`/discussions/[naddr]/audit` と `/discussions/[naddr]`・`/discussions/[naddr]/approve` は、同じ会話の承認状態を同じイベント集合から構築していない。そのため、監査画面で承認イベントを取得できても、他画面が別の最大3 relay に問い合わせて当該イベントを取得できなければ、投稿を未承認として表示できてしまう。

これは NIP-72 の解釈差ではなく、009 の relay 選別・キャッシュ再利用・投稿復元の画面間不統一による整合性欠陥である。

## 再現条件

1. naddr に複数の relay hint があり、その先頭3件に承認イベントがない、または応答しない。
2. 設定済み read relay のいずれかには同じ承認イベントがある。
3. `/audit` を開くと、relay hint を渡さないため設定 relay が選ばれ、投稿と承認イベントを合わせて取得できる。
4. `/` または `/approve` を開くと、hint 優先の3件だけを選び、承認イベントを得られない。

このとき監査は「承認済み」、詳細・承認は「未承認」を表示し得る。

## 根拠

| 経路 | 投稿の入力 | 承認の入力 | relay 候補の実際の入力 |
|---|---|---|---|
| 詳細 `/[naddr]` | 承認イベント `content` から復元 | kind 4550 の完了待ち read | naddr hint + configured。成功実績は未使用 |
| 承認 `/[naddr]/approve` | kind 1/1111 のストリーム | kind 4550 の別ストリーム | naddr hint + configured。成功実績は未使用 |
| 監査 `/[naddr]/audit` | kind 1/1111/4550 の混在ページ | 同一ページ内の kind 4550 | configured のみ。naddr hint と成功実績は未使用 |

- `relay-candidate-selector.ts` は `hint → recommended → successful → configured → default` の順で、3件に達した時点で後続候補を捨てる。
- 詳細画面は `successfulRelays` を cache に保存する一方、その後の候補選別へ渡していない。
- 監査ページは cache の `successfulRelays` を保存する一方、その後の候補選別へ渡していない。
- 監査 read の kind 混在 filter に `limit: 10` を付けているため、ページの投稿とその承認イベントが別ページに分断される危険もある。これは今回の逆転表示を増幅し得る別の欠陥である。

## 修正方針

1. 投稿・承認の状態集約を画面ではなく共通の Discussion moderation read に寄せる。同じ会話・同じ更新世代では、詳細、承認、監査が同じ投稿・承認スナップショットを利用する。
2. 全経路で relay 候補の入力を統一する。naddr hint、会話推奨 relay、cache の成功実績、configured、default を必ず渡し、選択結果と選択根拠を観測可能にする。
3. 初回の3 relay 制約は維持する。ただし、承認状態が「未観測」のまま partial/timeout になった場合は未承認と確定表示せず、「承認情報を確認中」または「一部 relay の結果」と表示し、次候補へ限定再読取できるようにする。
4. 監査ページングは、表示対象の投稿・申請イベントを先に最大10件取得し、そのイベントIDを使って関連承認を別 read で取得する。承認イベントの時刻や kind 混在 filter の上限で状態関連が欠落しないようにする。
5. 承認済み/未承認という最終ラベルは、共通スナップショットの event ID による結合結果だけから導出する。各画面が独自に `PostApproval[]` を再構築して差異を作らない。

## 実装順序

1. 失敗テストを追加する。hint relay に承認がなく configured/successful relay にだけ承認がある条件で、詳細・承認・監査が同じ「承認済み」状態になることを検証する。
2. 共通の read 入力・結果・状態ラベルを型として定義し、候補選別の呼び出しを一箇所へ集約する。
3. 詳細と承認画面を共通 moderation read へ移行する。既知イベントを暫定表示しても、relay read の結果と event ID でマージする。
4. 監査を「主イベント10件 + その対象の承認照会」へ移行し、詳細・承認と同じ状態集約器を使用する。
5. partial と EOSE の表示契約を更新し、未観測を未承認として断定しない。
6. unit・画面統合・relay選別・ページングのテスト後、`npx tsc --noEmit`、`npm run lint`、`npm test`、`npm run build` を実行する。

## 受入条件

- 同一の投稿IDに対して、監査で取得済みの有効な kind 4550 承認がある場合、詳細と承認画面も同じスナップショットでは承認済みと表示する。
- hint 優先の上限により承認イベントを未取得の場合、partial/timeout 中の画面は未承認を確定表示しない。
- cache の成功 relay は詳細・承認・監査のすべての候補選別に渡される。
- audit の表示対象10件と、その対象の承認状態は、kind 混在 filter の偶然の時刻順に依存しない。
- relay が同一イベントを複数返しても、承認判定・監査項目・投稿表示は event ID ごとに1件である。

## 対象外

- NIP-72 の kind 4550、`a`/`e`/`p` tag の意味や、承認・撤回の権限モデルの変更。
- relay を全件同時に読む方式への回帰。初回は009の最大3 relay方針を維持する。
