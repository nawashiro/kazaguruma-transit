### User Story 1 - 会話を閲覧し投稿・評価する (Priority: P1)

一般ユーザーは会話一覧から対象会話を選び、承認済み投稿を閲覧し、投稿や賛否評価を行える。

**Why this priority**: 意見収集の中心機能であり、投稿と評価がなければ会話データが蓄積されないため。

**Independent Test**: 会話詳細ページにアクセスし、ログイン後に投稿し、承認後の投稿に対して賛否評価ができることを確認すれば価値を単独で検証できる。

**Acceptance Scenarios**:

1. **Given** 利用可能な会話一覧が存在する, **When** ユーザーが会話一覧を開く, **Then** 承認済みの会話が新しい順で表示される
2. **Given** ログイン済みユーザーが会話詳細を開いている, **When** 投稿内容を入力して送信する, **Then** 投稿は承認待ちとして記録される
3. **Given** 承認済み投稿が存在する, **When** ユーザーが賛成または反対を選択する, **Then** 評価が記録され同一投稿の再評価対象から除外される

---

### User Story 2 - 会話を作成し後から掲載申請する (Priority: P2)

ログイン済みユーザーは新しい会話を作成し、必要に応じて会話編集画面から会話一覧への掲載申請を実行できる。

**Why this priority**: 会話作成と公開範囲調整を分離し、作成者が会話準備後に掲載判断できるため。

**Independent Test**: 会話作成後に編集画面へ遷移し、掲載申請操作を行うと会話一覧向け申請が記録されることを確認できる。

**Acceptance Scenarios**:

1. **Given** ログイン済みユーザーが会話作成画面を開いている, **When** タイトル・説明を入力して作成する, **Then** 新規会話が作成され会話URLが表示される
2. **Given** 会話作成者が会話編集画面を開いている, **When** 掲載申請操作を実行する, **Then** 会話一覧に対する掲載申請が新規作成される

---

### User Story 3 - 全ユーザー公開の承認・監査画面で役割に応じて操作する (Priority: P3)

すべてのユーザーは会話詳細および会話一覧の承認画面・監査画面を閲覧でき、モデレーターまたは会話作成者のみが承認操作を実行できる。モデレーター昇格申請の申請・承認・確認は会話編集画面で行い、画面は全ユーザーが閲覧できる。権限のないユーザーは、操作不能であることをUIで認識でき、モデレーター昇格申請を会話作成者へ送信できる。

**Why this priority**: 可視性を全員に開放しつつ、実行権限を限定することで透明性と安全性を両立できるため。

**Independent Test**: 権限あり/なし双方で同じ画面を閲覧できること、権限なしでは操作不可表示になること、昇格申請が監査ログに記録されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 任意ユーザーが承認画面を開いている, **When** 画面を閲覧する, **Then** 承認待ち・承認済みの状態を閲覧できる
2. **Given** モデレーターまたは会話作成者が承認画面を開いている, **When** 承認待ち投稿を承認または撤回する, **Then** 承認状態が更新される
3. **Given** 権限のないユーザーが承認画面を開いている, **When** 承認操作領域を表示する, **Then** 操作不能であることがUIで示される
4. **Given** 権限のないユーザーが昇格を希望している, **When** 昇格申請を送信する, **Then** 申請イベントが監査ログに表示される
5. **Given** 会話作成者が昇格申請一覧を開いている, **When** 申請に対して承認済み/未承認の状態を更新する, **Then** 対象ユーザーの昇格状態が更新され結果が監査ログに表示される
6. **Given** 任意ユーザーが監査画面を開いている, **When** 初回表示する, **Then** 最新10件が表示され、追加読み込み操作でさらに過去10件が追記表示される
7. **Given** 権限のないユーザーが操作ボタンを視認している, **When** 画面を表示する, **Then** ボタンは無効状態で表示され、直下に操作不能理由が表示される
8. **Given** 任意ユーザーが会話編集画面を開いている, **When** モデレーター管理セクションを表示する, **Then** 現在のモデレーター一覧（Mnemonic code）と昇格申請ユーザー一覧を閲覧できる
9. **Given** 会話作成者が会話編集画面を開いている, **When** モデレーターの追加または削除を実行する, **Then** モデレーター集合が更新され監査ログへ記録される

---

### Edge Cases
- 無効な会話アドレスで詳細・監査・編集・承認画面にアクセスした場合は、対象不正メッセージを表示し一覧へ戻れること
- 未ログイン状態で署名が必要な操作（投稿、評価、作成、承認、撤回、編集、削除）を行うとログイン導線が表示されること
- 一覧用会話アドレスや必須設定が欠落している場合、対象機能へのナビゲーション自体が表示されないこと
- 承認イベント本文が不正JSONなどで復元不能な場合でも、画面全体が停止せず復元可能なイベントのみ表示すること
- 同一会話IDに複数更新版が存在する場合、最新版のみを有効会話として扱うこと
- 承認対象が未存在または承認者本人の承認が見つからない撤回操作は無変更で終了すること
- 監査ログの追加読み込み時に件数不足が発生した場合、残件のみ表示して重複表示しないこと
- 会話・承認・監査の各画面でデータ未取得時はローディング表示、0件時は空状態メッセージを表示すること

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは会話一覧ページで、一覧掲載として承認済みの会話のみを新しい順で表示しなければならない
- **FR-002**: システムは各会話を固有アドレスで識別し、会話詳細・監査・編集・承認画面へ遷移できなければならない
- **FR-003**: システムは会話詳細で、承認済み投稿のみを評価対象および表示対象として扱わなければならない
- **FR-004**: システムはログイン済みユーザーに投稿作成を提供し、投稿は会話への所属情報と任意タグ情報を保持しなければならない
- **FR-005**: システムはモデレーターまたは会話作成者による投稿承認を提供し、承認後に会話表示へ反映しなければならない
- **FR-006**: システムは承認者本人による承認撤回を提供し、撤回操作を NIP-09 `kind:5` deletion request として記録し、撤回後に会話表示から除外できなければならない
- **FR-007**: システムはユーザー評価（賛成/反対）を記録し、同一ユーザーが既に評価した投稿を再評価候補から除外しなければならない
- **FR-008**: システムは評価結果を集計し、合意形成分析（意見グループ分けと代表意見提示）を提供しなければならない
- **FR-009**: システムは会話作成画面でタイトル・説明を受け付け、入力制約を検証しなければならない
- **FR-010**: システムは会話作成と会話一覧への掲載申請を分離し、掲載申請は会話編集画面から会話作成者が実行できなければならない
- **FR-011**: システムは会話作成成功後に、会話URLと開始案内を利用者へ提示しなければならない
- **FR-012**: システムは会話作成時に内部識別子としてUUIDを採用し、ユーザーにID入力を要求せず、会話編集画面自体は全ユーザーに公開しつつ、編集・削除操作は会話作成者のみに提供しなければならない
- **FR-013**: システムは会話一覧管理画面を提供し、会話一覧に対する承認申請を閲覧でき、モデレーターまたは会話作成者のみが承認・撤回操作できなければならない。撤回は NIP-09 `kind:5` deletion request として扱わなければならない
- **FR-014**: システムは会話一覧監査画面と会話詳細監査画面を明確に分離し、前者は会話一覧掲載申請および昇格申請の履歴、後者は個別会話（投稿・昇格申請）に関わる履歴を責務として表示しなければならない。監査画面の承認済み/未承認表示はイベント種別を増やさず、コンテンツ修飾として表示しなければならない
- **FR-015**: システムは表示名を一切使用せず、すべての表示箇所でユーザー公開鍵から算出したBIP39日本語Mnemonic codeを表示しなければならない。表示は先頭3フレーズのみとし、Passkey作成時の名前入力は端末内ラベル用途に限定し送信してはならない
- **FR-016**: システムは未ログイン状態の署名必須操作でログイン/新規登録導線を表示しなければならない
- **FR-017**: システムは権限のないユーザーに対して、対象操作ボタンを非表示にせず無効状態で表示し、各ボタン直下に理由文を表示しなければならない
- **FR-018**: システムは会話一覧機能・会話機能を設定により有効/無効切替でき、無効時は対象機能へのナビゲーションを表示してはならない
- **FR-019**: システムは承認画面と監査画面をすべてのユーザーに公開しなければならない
- **FR-020**: システムは権限のないユーザーが会話作成者へモデレーター昇格申請を送信できるようにし、昇格申請イベントを `kind:1111` かつ `a`(対象会話), `p`(作成者公開鍵), `t=moderator-request` タグを持つ形式で記録し、監査ログ対象イベントとして通常投稿と区別して扱わなければならない
- **FR-021**: システムは監査ログを最新10件単位で取得・表示し、利用者の操作で過去10件ずつ追加取得できなければならない
- **FR-022**: システムはモデレーター昇格申請の承認済み/未承認状態の更新操作を会話作成者のみに許可し、結果を監査ログへ記録しなければならない
- **FR-023**: システムは会話一覧、会話詳細、承認画面、監査画面、会話一覧管理画面の各画面で、ローディング状態・空状態・エラー状態を識別可能な文言で表示しなければならない
- **FR-024**: システムは監査ログ画面で「さらに過去10件を表示」操作を提供し、追加取得可能な間は操作可能、残件がない場合は操作不能状態を表示しなければならない
- **FR-025**: システムは会話一覧画面と会話詳細画面のナビゲーションを明確に区別し、各画面から対応する監査画面へ一意に遷移できるUI導線を提供しなければならない
- **FR-026**: システムは会話編集画面のモデレーター管理セクションで、現在モデレーターをMnemonic codeで一覧表示し、昇格申請ユーザー一覧を表示しなければならない
- **FR-027**: システムは会話編集画面において、会話作成者のみにモデレーター追加・削除操作を許可し、権限のないユーザーには操作不能UIを表示しなければならない
- **FR-028**: システムは監査画面で承認済み状態を表示する際、承認者の公開鍵から導出したBIP39日本語Mnemonic code（先頭3フレーズ）をコンテンツ修飾として表示しなければならない

### Assumptions

- 本仕様は2026-02-25時点の既存コードと`docs/discussion`配下ドキュメントを正として整理する
- FR-008（合意形成分析）は既存実装（`evaluation-service`/`polis-consensus`）を再利用し、仕様化対象では挙動固定・回帰確認を基本とする
- イベント配信・保存は分散リレー前提であり、単一リレー障害時にも他リレーでの読取継続を前提とする
- ログイン済み判定は端末保存済み認証情報を利用できることを前提とする
- 会話一覧への掲載可否は一覧管理権限者の承認判断に委ねられる
- BIP39 Mnemonic codeの表示は公開鍵から決定的に導出され、同一公開鍵で常に同一表記となる。表示言語は日本語、表示長は先頭3フレーズに固定する

### Key Entities *(include if feature involves data)*

- **会話**: 内部UUID識別子、タイトル、説明、作成者、モデレーター集合、作成時刻を持つ議題単位。一覧掲載申請の参照先となる
- **投稿**: 会話に紐づく本文意見。`kind:1111` を基本とし、後方互換として `kind:1` 読み取りを許容する。投稿者、会話参照、任意タグ、投稿時刻を持ち、承認されるまで公開対象外
- **投稿承認**: 投稿を会話内で公開可能とする承認記録。対象投稿、承認者、承認時刻を持つ
- **投稿評価**: 投稿に対する賛否表明。評価者、対象投稿、評価値、評価時刻を持つ
- **一覧掲載申請**: 作成済み会話を会話一覧へ載せるための申請記録。`kind:1111` で表現し、`a`(一覧参照) と `q`(対象会話参照) を含む。申請者、対象会話参照、申請時刻を持つ
- **モデレーター昇格申請**: 一般ユーザーが会話作成者へ送る昇格希望申請。`kind:1111` のうち `t=moderator-request` を持つイベントとして表現し、`a` と `p` タグで対象会話と宛先作成者を示す
- **会話一覧監査ログ項目**: 会話一覧監査画面で扱う履歴。掲載申請と昇格申請の requested イベントを時系列表示し、承認済み/未承認は承認者Mnemonic codeを含むコンテンツ修飾で示す
- **会話詳細監査ログ項目**: 会話詳細監査画面で扱う履歴。投稿 submitted と昇格申請 requested を時系列表示し、承認済み/未承認は承認者Mnemonic codeを含むコンテンツ修飾で示す

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新規ユーザーが会話作成画面で入力開始してから会話URL取得まで、90%以上のケースで5分以内に完了できる
- **SC-002**: 承認権限者が承認待ち投稿を承認してから会話詳細表示に反映されるまで、95%以上のケースで10秒以内に確認できる
- **SC-003**: 監査ログ画面で対象期間の主要操作（投稿、承認、掲載申請、昇格申請）を100%追跡できる
- **SC-004**: 権限外ユーザーの操作試行時に、100%で操作不能UIが表示される
- **SC-005**: 未ログイン状態で署名必須操作を実行した場合、100%でログイン/新規登録導線が表示される
- **SC-006**: 監査ログの初回表示で最新10件を表示し、追加読み込み1回ごとに過去10件が重複なく追加される
- **SC-007**: 昇格申請に対する承認済み/未承認状態の更新操作の100%が会話作成者によってのみ実行され、結果が監査ログで追跡できる
- **SC-008**: 権限のないユーザーが表示対象画面で操作ボタンを閲覧した場合、100%で無効状態表示と理由文が同時に提示される
- **SC-009**: 会話一覧、会話詳細、承認、監査、一覧管理の各画面で、ローディング/空/エラーの各状態が100%で表示される
