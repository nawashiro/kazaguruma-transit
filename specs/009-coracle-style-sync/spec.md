**Created**: 2026-07-09

**Status**: Draft

**Input**: User description: "今回は009仕様を作りたいです。Coracle式の通信高速化を導入したいです。詳細は知識にあります。"

> **Note**: この仕様は `.specify/memory/constitution.md` と `AGENTS.md` の原則に準拠して作成されています。Coracle / Welshman の調査知識 `[[references/coracle-social-performance-critical-reread-2026]]` と `[[references/coracle-welshman-performance-architecture]]` を背景情報として参照し、kazaguruma-transit の Discussion / Nostr 体験に合わせて仕様化します。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 会話画面を待たされずに開く (Priority: P1)

利用者は会話一覧、会話詳細、承認画面、監査画面を開いたとき、全リレーの完了を待たされず、取得できた内容から段階的に画面を利用できる。

**Why this priority**: Discussion 機能の主要な不満は、分散 relay の遅延が画面全体の利用開始を妨げることにある。最初の表示が速くならなければ、他の最適化の価値を体感しにくい。

**Independent Test**: 複数 relay のうち一部が遅延または無応答の状態で会話詳細を開き、利用可能な投稿・承認・状態表示が先に表示され、遅延 relay の完了を待たずに画面操作を始められることを確認する。

**Acceptance Scenarios**:

1. **Given** 一部の relay が応答し、別の relay が遅延している, **When** ユーザーが会話詳細を開く, **Then** 応答済み relay から得た会話情報と投稿候補が先に表示され、未完了の取得状態が画面上で識別できる
2. **Given** 会話メタデータは取得済みだが投稿または承認の取得が継続中である, **When** ユーザーが画面を閲覧する, **Then** 会話タイトル、ナビゲーション、投稿作成導線は利用可能で、未取得領域だけが読み込み状態になる
3. **Given** relay 応答が idle-timeout または hard-timeout で完了した, **When** 画面が結果を表示する, **Then** Not Found 固定表示ではなく、部分取得であること、再読み込み可能であること、表示中の内容が暫定であることが日本語で示される

---

### User Story 2 - 必要な relay だけから取得する (Priority: P2)

利用者は会話、投稿、承認、評価、監査ログなど目的の異なる情報を開くとき、画面目的に合う relay へ絞られた問い合わせにより、不要な relay ブロードキャストによる遅延を受けにくくなる。

**Why this priority**: Coracle式高速化の中核は「全体同期」ではなく「選別された部分同期」である。画面ごとに必要なイベント集合と relay 集合を縮小できれば、体感速度と通信量の双方を改善できる。

**Independent Test**: 会話一覧、会話詳細、監査ログの各画面で発行される取得要求を観測し、画面目的ごとに対象イベント種別、タグ条件、取得件数、relay 候補が過剰でないことを確認する。

**Acceptance Scenarios**:

1. **Given** 会話一覧を開く, **When** システムが Nostr イベントを取得する, **Then** 一覧表示に必要な会話掲載・承認関連イベントに範囲を限定し、個別会話の全文投稿取得を同時に行わない
2. **Given** 個別会話詳細を開く, **When** システムが投稿と承認を取得する, **Then** 対象会話に紐づくイベントへ範囲を限定し、無関係な会話イベントを混在させない
3. **Given** 監査ログで「さらに過去10件を表示」を実行する, **When** 追加取得を行う, **Then** 既取得範囲より過去の監査対象イベントだけを最大10件取得し、既取得データの再スライスで代替しない

---

### User Story 3 - 重複や再描画で画面が重くならない (Priority: P3)

利用者は同一イベントが複数 relay から届く場合でも、同じ投稿・承認・監査項目が重複表示されず、画面更新が過剰に発生しない安定した表示を得られる。

**Why this priority**: relay を複数使う Nostr では重複イベントが自然に発生する。重複排除と更新頻度制御がないと、取得速度を上げても UI が不安定になり、アクセシビリティと操作性が悪化する。

**Independent Test**: 同一イベントを複数 relay から到着させ、表示件数、監査ログ、承認状態、読み上げ対象のステータス表示が重複せず、短時間に過剰更新されないことを確認する。

**Acceptance Scenarios**:

1. **Given** 同一投稿が複数 relay から届く, **When** 会話詳細がイベントを反映する, **Then** 投稿は1件として表示され、取得元 relay の差によって重複表示されない
2. **Given** 同一承認イベントが複数 relay から届く, **When** 承認状態を再構築する, **Then** 承認済み判定と監査ログ項目は重複せず、同じ承認が複数回数えられない
3. **Given** 短時間に多数のイベントが到着する, **When** UI が状態を更新する, **Then** 利用者の操作中のボタン、入力欄、フォーカス位置が不必要に失われない

---

### User Story 4 - 再訪問時に既知データを活かす (Priority: P4)

再訪問ユーザーは、前回取得済みの会話メタデータ、既知イベント、relay 応答状況を活用して、初回より少ない待ち時間で画面を確認できる。

**Why this priority**: Coracle式高速化はキャッシュだけが主因ではないが、既知データの再利用は再訪問時の体感速度に効く。新鮮さを損なわない範囲で局所記憶を使う必要がある。

**Independent Test**: 同じ会話を初回表示後に再読み込みし、既知データが暫定表示として使われ、新規 relay 取得結果により更新・補完されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザーが過去に会話メタデータを取得済みである, **When** 同じ会話を再訪問する, **Then** 既知の会話タイトルやモデレーター情報を暫定表示し、relay 取得結果で更新できる
2. **Given** 既知イベントと新規イベントが混在する, **When** システムが画面を再構築する, **Then** 既知イベントは重複なく再利用され、新規イベントだけが追加または更新として扱われる
3. **Given** relay からより新しい会話定義または承認状態が届く, **When** 暫定表示と差分がある, **Then** 表示は新しい状態へ更新され、古い状態を確定情報として残さない

### Edge Cases

- 全 relay が無応答または接続不能の場合、画面は停止せず、取得不能状態、再試行導線、既知データの有無を表示する
- relay が EOSE を返さない場合でも、idle-timeout / hard-timeout により読み込みを完了扱いにし、完了理由を保持する
- relay がイベントをソート順どおり返さない場合でも、表示順はイベント時刻と仕様上の優先順位で決定し、到着順に依存しない
- 同一イベントIDが複数 relay から届いた場合、表示・集計・監査では1件として扱う
- 同一会話定義に複数版が存在する場合、最新版を有効とし、旧版は必要な履歴表示以外で優先しない
- 取得上限より多いイベントが存在する場合、初回表示では画面目的に必要な件数だけを取得し、追加読み込みで過去方向に進む
- 部分取得状態でユーザーが投稿・承認・評価を行う場合、操作対象が現在表示中の有効イベントに基づくことを確認し、不整合の可能性がある場合は再取得を促す
- キャッシュ済みデータが古い場合、relay 取得結果で補正し、古いデータだけで Not Found や承認不可を確定しない
- アクセシビリティ支援技術に対して、読み込み中、部分取得、取得完了、エラーの状態変化が理解できる文言と状態属性を提供する

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは Discussion 関連画面の Nostr 取得を、画面目的ごとの取得単位に分離しなければならない
- **FR-002**: システムは会話一覧、会話詳細、承認画面、監査画面、会話編集画面ごとに、取得対象イベント種別、タグ条件、件数上限を明確に定義しなければならない
- **FR-003**: システムは各取得単位で、全 relay の完全応答を待たずに利用可能な結果を画面へ反映できなければならない
- **FR-004**: システムは取得完了理由として、少なくとも完全応答、応答停滞、最大待機時間到達、キャンセルを区別して保持しなければならない
- **FR-005**: システムは応答停滞または最大待機時間到達で完了した場合、Not Found と断定せず、部分取得状態としてユーザーに表示しなければならない
- **FR-006**: システムは画面目的に応じて、問い合わせ先 relay 候補を少数に選別し、不要な全 relay 同時問い合わせを避けなければならない
- **FR-007**: システムは relay 選別時、会話定義に含まれる推奨 relay、イベントの relay hint、利用者設定、既知の成功実績、既定 relay を考慮しなければならない
- **FR-008**: システムは relay 選別の結果、候補が不足する場合でも、既定 relay または設定済み relay にフォールバックし、取得不能で停止してはならない
- **FR-009**: システムは初回取得で使用する relay 数に上限を設け、画面ごとの必要性がない限り同時接続数を増やしてはならない
- **FR-010**: システムは同一イベントIDを複数 relay から受信した場合、表示、集計、承認判定、監査ログでは1件として扱わなければならない
- **FR-011**: システムは重複イベントを検出した場合でも、後続の relay hint または取得元情報として有用な情報は保持できなければならない
- **FR-012**: システムは既知イベント、会話メタデータ、承認状態、relay 応答実績を、再訪問時の暫定表示または取得優先度判断に利用できなければならない
- **FR-013**: システムはキャッシュまたは既知データを表示する場合、relay 取得結果で更新・訂正できる暫定情報として扱わなければならない
- **FR-014**: システムは会話フィードの新鮮さを守るため、既知データだけで最新投稿一覧を確定せず、relay 取得による補完を試みなければならない
- **FR-015**: システムはメタデータや設定に近い情報では、既知データを優先的に表示し、relay 取得結果で差分更新できなければならない
- **FR-016**: システムは複数の取得要求が短時間に発生した場合、同じ relay と近い条件への重複問い合わせを抑制しなければならない
- **FR-017**: システムは監査ログの初回表示と追加読み込みで、実際の relay 取得件数に最大10件の制約を適用しなければならない
- **FR-018**: システムは追加読み込み時、既取得範囲より過去の未取得イベントを対象に再問い合わせし、既取得イベント集合の表示切替だけで代替してはならない
- **FR-019**: システムはイベント到着順に依存せず、投稿、承認、評価、監査ログの表示順を仕様上の並び順で安定させなければならない
- **FR-020**: システムは短時間に多数のイベントが到着した場合、利用者が認識できないほど頻繁な画面更新を避け、操作中のフォーカスや入力内容を維持しなければならない
- **FR-021**: システムは取得中、部分取得、取得完了、取得不能、再試行可能の各状態を日本語で表示しなければならない
- **FR-022**: システムは状態表示について、スクリーンリーダー利用者にも状態変化が伝わるアクセシブルな表現を提供しなければならない
- **FR-023**: システムは relay 遅延、重複排除、部分取得、追加読み込み、再訪問時の既知データ利用を、テストまたは観測可能なログで検証できなければならない
- **FR-024**: システムは高速化により、投稿作成、承認、評価、モデレーター管理、監査の権限判定を弱めてはならない
- **FR-025**: システムは高速化により、NIP-72 / NIP-25 / NIP-09 / NIP-01 に基づく既存イベント解釈を変更してはならない
- **FR-026**: システムは relay 選別や既知データ利用が原因で、承認済み投稿を未承認として確定表示したり、未承認投稿を承認済みとして確定表示したりしてはならない
- **FR-027**: システムはユーザー操作で明示的に再読み込みできる導線を、部分取得または取得不能状態の画面に提供しなければならない
- **FR-028**: システムは取得戦略の設定値を、通常利用者に複雑な設定負担を与えず、保守者が調整可能な形で管理しなければならない
- **FR-029**: システムは高速化後も既存の Discussion 機能仕様で定義された表示名禁止、Mnemonic code 表示、権限外操作の無効表示、ローディング/空/エラー状態表示を維持しなければならない

### Key Entities *(include if feature involves data)*

- **取得単位**: 画面目的ごとに定義される Nostr イベント取得のまとまり。対象画面、イベント種別、タグ条件、件数上限、完了条件を持つ
- **Relay候補**: 取得単位に対して問い合わせ対象となる relay の候補集合。推奨 relay、relay hint、利用者設定、既知の成功実績、既定 relay から構成される
- **取得結果**: イベント集合、取得元 relay、完了理由、取得開始・完了時刻、部分取得かどうかを持つ結果
- **既知イベント記録**: 過去に受信したイベントIDと取得元 relay の対応。重複排除、relay hint 補強、再訪問時の暫定表示に使う
- **既知メタデータ**: 会話定義、モデレーター集合、ユーザー関連メタデータなど、再訪問時に暫定表示しやすい情報
- **部分取得状態**: relay 応答が完了しきっていないが画面利用を開始できる状態。完了理由、再試行可否、暫定表示の有無を持つ
- **取得観測ログ**: テストや保守確認のために、取得単位、relay 数、イベント数、重複数、完了理由、経過時間を確認できる記録

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一部 relay が3秒以上遅延する条件でも、会話詳細の主要情報が95%以上の試行で2秒以内に利用可能状態になる
- **SC-002**: 会話一覧、会話詳細、承認画面、監査画面の初回取得で、画面目的外のイベント取得がテストまたはログ上0件であることを確認できる
- **SC-003**: 同一イベントが3つの relay から到着しても、投稿、承認、監査ログの表示件数と集計件数は100%重複しない
- **SC-004**: relay が EOSE を返さない条件でも、100%の取得で完了理由が完全応答以外として記録され、ユーザーに部分取得または取得不能として表示される
- **SC-005**: 監査ログの初回表示と追加読み込みで、各取得操作が最大10件の実取得制約を守ることをテストで検証できる
- **SC-006**: 再訪問時、既知メタデータがある会話では、90%以上の試行でタイトルまたは基本情報が1秒以内に表示される
- **SC-007**: 高速化後も、投稿作成、承認、撤回、評価、モデレーター管理の既存権限テストが100%通る
- **SC-008**: 部分取得、取得不能、再試行可能の各状態について、画面上の日本語文言とアクセシブルな状態通知を100%確認できる
- **SC-009**: 連続イベント到着時に、入力中フォームの値とフォーカスが失われないことを主要 Discussion 画面で確認できる

## Assumptions

- 対象は kazaguruma-transit の Discussion / Nostr 関連画面であり、乗換検索やGTFS処理は本仕様の対象外とする
- Coracle / Welshman の設計は参考情報であり、外部ライブラリの全面導入を必須とはしない
- 高速化の核は「relay 数を減らす」単独施策ではなく、画面目的に合わせた選別された部分同期である
- 初期既定では、同時問い合わせ relay 数は少数に制限する。ただし relay 候補不足時は既定 relay へフォールバックする
- Discussion の既存イベント形式、権限モデル、監査ログ責務、Mnemonic code 表示方針は `008-document-discussion-spec` を引き継ぐ
- 既存実装には completion-aware read と streaming 系処理が一部存在するため、本仕様はそれらを統合・拡張する前提で計画される
- キャッシュや既知データは利便性向上のための暫定材料であり、relay 取得結果より常に強い確定情報として扱わない
- ユーザー向けには複雑な relay 戦略を設定させず、通常は自動選別で動作する
