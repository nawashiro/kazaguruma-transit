# Feature Specification: 会話編集画面UX改善

**Feature Branch**: `011-discussion-edit-ux`

**Created**: 2026-07-11

**Status**: Draft

**Input**: User description: "会話編集画面のスクリーンショットと要素調査で抽出した、モバイル横スクロール、固定ルビボタンの重なり、未ログイン時の導線不足、権限メッセージや戻る導線の重複、編集画面の情報過多、会話未検出の誤表示などを改善する仕様011を作成する。"

## Clarifications

### Session 2026-07-11

- Q: 未ログイン時のログイン導線をどこに置くか → A: 権限メッセージ内に明確な「ログイン」ボタンを1つ表示する。
- Q: 削除操作をどこに配置するか → A: 編集操作から分離した「危険な操作」セクションに置き、確認を必須にする。
- Q: 「編集」タブを誰に表示するか → A: すべてのユーザーに表示し、権限がない場合は説明とログイン導線を表示する。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - モバイルで会話を編集する (Priority: P1)

会話作成者は、スマートフォンの狭い画面でも編集フォームと操作ボタンを確認し、横スクロールせずに目的の操作を完了できる。

**Why this priority**: 現状は操作ボタンが画面外へはみ出し、主要な編集操作を妨げているため。

**Independent Test**: 幅390pxの画面で編集ページを開き、ページ全体と各操作ボタンが横方向にはみ出していないこと、フォームの全項目を縦方向のスクロールだけで確認できることを検証する。

**Acceptance Scenarios**:

1. **Given** 幅390pxの画面で編集ページを開いている、**When** ページを左右に移動せずにフォームを閲覧する、**Then** ページの表示幅は画面幅を超えず、編集カードと操作ボタンが画面内に収まる。
2. **Given** 編集操作ボタンが3つ表示されている、**When** 画面幅が狭い、**Then** ボタンは縦積みまたは画面内で折り返され、ラベルが切れず、各ボタンを個別に押せる。
3. **Given** 説明欄または申請理由欄に入力している、**When** 入力欄の右下や周囲を操作する、**Then** ルビ切替コントロールが入力欄・リサイズ操作・入力内容に重ならない。

---

### User Story 2 - 未ログイン状態から編集を開始する (Priority: P1)

未ログインユーザーは、編集できない理由を理解し、ログインまたはログイン方法へ迷わず進める。

**Why this priority**: 現状は操作が無効化されている一方で、ページ内にログインへの明確な導線がなく、次の行動が分からないため。

**Independent Test**: 未ログイン状態でページを開き、編集フォーム・操作ボタン・権限説明を確認して、ログイン導線から認証開始画面へ到達できることを検証する。

**Acceptance Scenarios**:

1. **Given** 未ログインユーザーが編集ページを開いている、**When** 編集権限が必要な領域を確認する、**Then** 「ログインが必要」であることと、権限メッセージ内の1つの「ログイン」ボタンが同じ文脈で表示される。
2. **Given** 未ログイン状態で保存・掲載申請・削除などの操作を見ている、**When** ログイン導線を選択する、**Then** 認証開始UIが開き、元の編集ページの状態が失われない。
3. **Given** ログイン済みだが会話作成者ではないユーザーがページを開いている、**When** 編集操作を確認する、**Then** 編集できない理由が「会話作成者のみ」と表示され、未ログイン向けの説明とは混同されない。

---

### User Story 3 - 編集画面の情報を理解して操作する (Priority: P2)

ユーザーは、会話情報の編集、会話一覧への掲載申請、削除、モデレーター管理の区別を理解し、必要な操作を短時間で見つけられる。

**Why this priority**: 現状は異なる目的と危険度の操作が一つのカードに連続しており、重複した戻る導線や権限メッセージが認知負荷を高めているため。

**Independent Test**: 編集ページを初見のユーザーに提示し、各セクションの目的、危険操作、戻る先を説明できることと、意図したセクションへ直接到達できることを検証する。

**Acceptance Scenarios**:

1. **Given** 編集ページを表示している、**When** ページ上部から内容を読む、**Then** 編集、掲載申請、モデレーター管理、「危険な操作」の区切りと目的が明確に分かれる。
2. **Given** 会話詳細ページ、監査ログページ、編集ページのいずれかを表示している、**When** ページ上部のタブナビゲーションを確認する、**Then** すべてのユーザーに「会話」「監査ログ」「編集」の3タブが表示され、現在のページだけが選択状態になる。
3. **Given** 内部的な会話IDを表示する必要がある、**When** ユーザーが会話IDを確認する、**Then** それが編集対象を識別するための読み取り専用情報であることが分かり、必要に応じて正確に扱える。

---

### User Story 4 - データ取得状態を正しく理解する (Priority: P2)

ユーザーは、会話データの読み込み中・取得遅延・会話不存在を区別でき、正常な会話を誤って「見つかりません」と判断しない。

**Why this priority**: 外部データの応答待ちと不存在が混同されると、編集ページから離脱したり、再試行の必要性を誤認したりするため。

**Independent Test**: 読み込み中、遅延、データ取得成功、会話不存在の各状態を再現し、状態ごとに異なる説明・操作が表示されることを検証する。

**Acceptance Scenarios**:

1. **Given** 会話情報の取得が完了していない、**When** ページを表示する、**Then** 読み込み中であることが表示され、「会話が見つかりません」とは表示されない。
2. **Given** 取得が遅延またはタイムアウトした、**When** ユーザーが状態を確認する、**Then** 遅延・再試行の案内が表示され、存在しない会話と区別できる。
3. **Given** 会話が確実に存在しない、**When** 取得処理が完了する、**Then** 不存在の説明と会話一覧へ戻る操作が表示される。

### Edge Cases

- 画面幅が320px程度まで狭くなっても、本文・入力欄・主要ボタンが横にはみ出さない。
- 長い日本語のタイトル、説明、モデレーター識別子が表示されても、カード幅やボタンラベルを壊さず、必要な場合は自然に折り返す。
- 未ログイン、ログイン済み非作成者、作成者、現在のモデレーター、作成者兼モデレーターで、権限説明と操作可能状態が矛盾しない。
- ルビ表示のオン・オフを切り替えても、固定コントロールがフォームや画面端の重要操作を覆わない。
- 保存・掲載申請・削除・昇格申請の処理中に、二重送信や意図しない別操作が起きない。
- 読み込みが一時的に失敗して再試行した場合、古い会話情報と新しい状態が混在しない。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 編集ページは、対象とするデスクトップおよびモバイルの画面幅で、横スクロールを必要とせず利用できなければならない。
- **FR-002**: 編集ページは、狭い画面幅でも主要操作のラベルと操作対象を表示領域内に収めなければならない。
- **FR-003**: ルビ表示コントロールは、対象とするすべての画面幅で、フォーム、入力文字、リサイズ操作、固定ナビゲーション、その他の重要な内容を覆ってはならない。
- **FR-004**: 未ログインユーザーが認証必須の操作に遭遇した場合、ページは関連する権限メッセージ内に1つの明確な「ログイン」ボタンを提示しなければならない。
- **FR-005**: 認証必須のメッセージは、関連する操作群ごとに一度だけ表示し、未ログイン状態とログイン済みだが編集権限がない状態を区別しなければならない。
- **FR-006**: ページは、会話編集、掲載申請、モデレーター管理、「危険な操作」の目的と視覚的な階層を分離しなければならない。削除は「危険な操作」に含め、実行前の確認を必須とする。
- **FR-007**: 会話関連ページは、すべてのユーザーに共通の「会話」「監査ログ」「編集」タブを提供し、現在のページを選択状態で示さなければならない。編集権限がない場合もタブは隠さず、遷移先で理由と必要に応じたログイン導線を示す。
- **FR-008**: 編集ページは独立した「会話に戻る」リンクを表示せず、会話詳細への移動には「会話」タブを使用しなければならない。
- **FR-009**: 読み取り専用の会話識別子は用途を明示し、編集可能な入力欄と誤認されないようにしなければならない。
- **FR-010**: ページは、読み込み中、取得遅延、取得成功、確認済み不存在を、異なる利用者向け内容と操作で区別しなければならない。
- **FR-011**: 認証開始またはデータ取得の再試行を行った場合、ページは入力済みの値とユーザーの戻り先の文脈を保持しなければならない。
- **FR-012**: すべての操作コントロールは、アクセシブルな名前、キーボード操作時の視認可能なフォーカス表示、実用上可能な場合は44px以上のタッチ領域を備えなければならない。
- **FR-013**: ユーザー向けテキストは理解しやすい日本語とし、ルビを除くテキストはプロジェクトの最小14pxフォントサイズ要件を満たさなければならない。
- **FR-014**: ページは一貫した単一のメインコンテンツランドマークを公開し、支援技術で論理的に辿れる見出し階層を維持しなければならない。
- **FR-015**: 本機能は、編集、掲載申請、削除、モデレーター昇格、昇格審査に対する既存の認可ルールを維持しなければならない。

### Key Entities

- **編集対象会話**: タイトル、説明、会話識別子、作成者、モデレーターを持つ会話情報。
- **利用者権限状態**: 未ログイン、ログイン済み非作成者、作成者、モデレーターなど、各操作の可否と説明を決める状態。
- **会話取得状態**: 読み込み中、取得遅延、取得成功、確認済み不存在を表す表示状態。
- **編集操作グループ**: 会話情報変更、掲載申請、削除、モデレーター管理に分類された操作群。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 幅320px以上のモバイル画面で、編集ページの横方向オーバーフローを0件にする。
- **SC-002**: 幅390pxの画面で、保存・掲載申請・削除の主要操作をすべて画面内から実行できる状態にする。
- **SC-003**: ルビ表示のオン・オフいずれでも、固定ルビコントロールが入力欄または主要操作に重なるケースを0件にする。
- **SC-004**: 定義済みの未ログイン操作シナリオを実施した検証者全員が、編集タブから権限メッセージ内の「ログイン」ボタンへ1回以内の操作で到達できる状態にする。
- **SC-005**: 読み込み中・取得遅延・不存在の状態判別テストで、各状態を正しく識別できる割合を100%にする。
- **SC-006**: 定義済みの主要編集シナリオを、幅320px・390px・1440pxの各検証環境で、検証者の90%以上が初回操作で完了できる状態にする。
- **SC-007**: 主要操作、フォーム項目、状態メッセージをキーボード操作と支援技術で確認でき、重大なアクセシビリティ阻害を0件にする。
- **SC-008**: 既存の権限ルールと保存・申請・削除の動作を回帰テストで維持する。

## Assumptions

- 対象は指定された会話編集ページと、そのページに共通表示される会話タブ・ナビゲーション・ルビ切替コントロールである。
- デスクトップとモバイルのレスポンシブ表示を同一機能範囲で改善し、編集機能自体の権限モデルやデータ形式は変更しない。
- 認証機能は既存の仕組みを再利用し、今回の仕様では認証方式そのものを変更しない。
- 会話データの正本と取得経路は既存仕様を維持し、今回の対象は取得結果に応じた表示状態と再試行導線である。
- 画面文言は日本語を正本とし、既存のルビ表示機能との互換性を維持する。
- 本機能の検証では、少なくとも幅390pxのモバイル画面と幅1440pxのデスクトップ画面を対象とする。
