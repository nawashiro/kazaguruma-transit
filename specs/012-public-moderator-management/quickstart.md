# Quickstart: 公開モデレーター管理の検証

## 前提条件

- 開発用の会話データと、会話作成者・一般ユーザーのログイン状態を用意する。
- 開発サーバーを `npm run dev` で起動する。
- モデレーター申請イベントと複数のkind 34550イベントを用意できるテスト設定を利用する。

## 自動検証

1. `npm test -- --runInBand` を実行する。
2. `npm run lint` を実行する。
3. `npm run build` を実行する。

主に、申請状態の導出、kind 34550の時刻境界、許可・削除・直接追加の一括確定、権限別の画面表示、タブのキーボード操作を検証する。

## 手動確認

### 公開一覧

1. 未ログインで対象会話の「モデレーター」タブを開く。
2. モデレーターと申請中ユーザーが別の一覧に表示され、それぞれのニーモニックと完全なユーザーIDを確認できることを確認する。
3. 申請理由・日時が申請中一覧に表示され、申請位置にはログイン導線だけがあることを確認する。

### 申請状態の時刻境界

1. 現在のkind 34550より前の申請を用意し、申請中一覧に表示されないことを確認する。
2. 現在のkind 34550と同時刻および後の申請を用意し、申請中一覧に表示されることを確認する。
3. 会話作成者が変更を確定した後、確定前の申請が申請中一覧に戻らないことを確認する。

### 会話作成者の一括変更

1. 会話作成者でログインし、申請中ユーザーを許可対象、既存モデレーターを削除対象としてチェックする。
2. 直接追加するユーザーIDを入力し、確定前は公開一覧が変化せず、変更要約だけが表示されることを確認する。
3. 「変更を確定」を一度選択し、許可・直接追加・削除が同時に反映され、各利用者が二一覧の一方だけに表示されることを確認する。
4. 発行失敗を再現し、選択内容を保持したまま日本語のエラーと再試行操作が表示されることを確認する。

### 基本情報の分離と表示幅

1. 一般ユーザーと未ログインユーザーで、基本情報タブ・編集フォーム・危険な操作が表示されないことを確認する。
2. 会話作成者では、基本情報画面にモデレーター管理がなく、タイトル・説明・掲載申請・会話削除だけがあることを確認する。
3. 幅320px、390px、1440pxで、モデレーター画面と基本情報画面に横方向オーバーフローがないことを確認する。
