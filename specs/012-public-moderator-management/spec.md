# Feature Specification: 公開モデレーター管理

**Feature Branch**: `012-public-moderator-management`

**Created**: 2026-07-11

**Status**: Draft

**Input**: User description: "会話のモデレーター情報と申請状況を全利用者に公開し、会話作成者だけが基本情報を編集できるよう、会話編集画面を再構築する。モデレーター関連は公開ページとして分離し、モデレーター中のユーザーと申請中のユーザーの二分類に統一する。"

## Clarifications

### Session 2026-07-11

- Q: 申請を申請中一覧に表示する基準は何か。 → A: 現在の34550イベントの発行時刻を基準にし、その時刻以後に発行された申請だけを申請中として表示する。
- Q: 会話作成者は申請の許可とモデレーターの削除をどのように確定するか。 → A: 各対象をチェックボックスで選択し、最後に一つの確定ボタンでまとめて反映する。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 公開されたモデレーター状況を確認する (Priority: P1)

一般ユーザーは、ログインの有無にかかわらず、会話のモデレーターをしているユーザーと申請中のユーザーを確認し、運営体制と申請状況を理解できる。

**Why this priority**: モデレーターの選任状況は会話の信頼性に関わる公開情報であり、管理画面に閉じ込めないことが本機能の中心価値である。

**Independent Test**: 未ログイン状態でモデレーター画面を開き、両方の一覧と各ユーザーの識別情報が読めることを確認する。

**Acceptance Scenarios**:

1. **Given** 会話にモデレーターがいる、**When** 任意の利用者がモデレーター画面を開く、**Then** モデレーターをしている全ユーザーが表示される。
2. **Given** 会話に保留中のモデレーター申請がある、**When** 任意の利用者がモデレーター画面を開く、**Then** 申請中の全ユーザー、申請日時、申請理由が表示される。
3. **Given** 一人のユーザーがモデレーターをしている、**When** そのユーザーの行を見る、**Then** ニーモニックとユーザーIDが同じ行または同じカード内で確認できる。
4. **Given** 申請もモデレーターもいない会話、**When** 任意の利用者がモデレーター画面を開く、**Then** 両方の一覧が空であることを区別して表示する。

---

### User Story 2 - モデレーターを申請する (Priority: P1)

ログイン済みの一般ユーザーは、公開モデレーター画面から申請理由を添えてモデレーターへの就任を申請し、自分の申請が公開の申請中一覧に反映されたことを確認できる。

**Why this priority**: 申請導線を、公開される選任情報と同じ文脈に置くことで、一般ユーザーが申請の意味と現在の状態を理解できる。

**Independent Test**: ログイン済みでモデレーターではない利用者として画面を開き、申請を送信して申請中一覧に自分が一度だけ表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** ログイン済みで、会話作成者でもモデレーターでもない利用者、**When** 任意の申請理由で申請する、**Then** 申請が送信され、申請中のユーザーとして表示される。
2. **Given** 未ログインの利用者、**When** モデレーター画面を開く、**Then** 一覧は閲覧でき、申請フォームの代わりに文脈に沿ったログイン導線が表示される。
3. **Given** すでに申請中の利用者、**When** モデレーター画面を再び開く、**Then** 重複申請の送信操作は提示されず、自分の申請中状態を確認できる。
4. **Given** すでにモデレーターをしている利用者または会話作成者、**When** モデレーター画面を開く、**Then** 自分の昇格申請操作は提示されない。

---

### User Story 3 - 会話作成者がモデレーターを管理する (Priority: P1)

会話作成者は、公開モデレーター画面で申請中ユーザーの許可と既存モデレーターの削除をチェックボックスで選択し、ユーザーIDによる直接追加を含む変更内容を最後に一度だけ確定できる。

**Why this priority**: 選任の公開性を保ちながら、会話作成者が必要な運営操作を一箇所で安全に行える必要がある。

**Independent Test**: 会話作成者として、許可する申請者と削除する既存モデレーターを選択し、直接追加も入力したうえで、確定前には公開一覧が変化せず、確定後に二つの公開一覧が矛盾なく変化することを確認する。

**Acceptance Scenarios**:

1. **Given** 会話作成者と申請中ユーザー、**When** 会話作成者が申請者を許可対象として選択して変更を確定する、**Then** そのユーザーは申請中一覧から消え、モデレーターをしているユーザー一覧に一度だけ表示される。
2. **Given** 会話作成者、**When** 有効なユーザーIDを直接追加対象として入力し、変更を確定する、**Then** 対象ユーザーはモデレーターをしているユーザー一覧に表示される。
3. **Given** 会話作成者と既存モデレーター、**When** 会話作成者が削除対象として選択して変更を確定する、**Then** 対象ユーザーはモデレーターをしているユーザー一覧から除かれる。
4. **Given** 会話作成者が許可・直接追加・削除のいずれかを選択または入力している、**When** まだ変更を確定していない、**Then** 公開一覧の現在の状態は変化せず、確定される変更内容を確認できる。
5. **Given** 会話作成者ではない利用者、**When** モデレーター画面を開く、**Then** 公開一覧と自身に許可された申請操作だけが表示され、許可・直接追加・削除の選択操作は表示されない。
6. **Given** 許可済みの申請、**When** 会話作成者がモデレーター画面を確認する、**Then** その申請に対する許可操作は再表示されない。

---

### User Story 4 - 会話作成者だけが基本情報を管理する (Priority: P2)

会話作成者は、モデレーター管理と切り離された基本情報画面で、会話のタイトル・説明、掲載申請、会話削除を管理できる。一般ユーザーは同じ情報を会話ページで閲覧できるため、基本情報の管理画面を見ない。

**Why this priority**: 公開済みの会話情報と運営体制を混在させず、編集権限のない利用者に操作不能なフォームを見せないためである。

**Independent Test**: 会話作成者と一般ユーザーでタブおよび直接URLを確認し、作成者だけが基本情報の管理内容に到達できることを検証する。

**Acceptance Scenarios**:

1. **Given** 会話作成者、**When** 会話関連のナビゲーションを確認する、**Then** 会話、監査ログ、モデレーター、基本情報の編集へ移動できる。
2. **Given** 会話作成者ではない利用者、**When** 会話関連のナビゲーションを確認する、**Then** 会話、監査ログ、モデレーターへ移動できるが、基本情報の編集への入口は表示されない。
3. **Given** 会話作成者ではない利用者、**When** 基本情報の管理URLを直接開く、**Then** 編集フォームや危険な操作は表示されず、会話ページへ戻る導線が示される。
4. **Given** 会話作成者、**When** 基本情報画面を開く、**Then** タイトル・説明の編集、掲載申請、会話削除だけが目的別に表示され、モデレーター管理は表示されない。

### Edge Cases

- 同一ユーザーが複数回申請しても、申請中一覧では同一人物を重複表示せず、利用者が理解できる一つの現在の申請として扱う。
- モデレーターをしているユーザーは申請中一覧に同時表示されない。
- 現在の34550イベントより前に発行された申請は、申請中一覧に表示しない。同じ発行時刻の申請は、取りこぼしを防ぐため申請中として扱う。
- 会話作成者がモデレーターを削除した後、過去の承認済み申請が新しい保留申請として自動復活しない。
- 許可、直接追加、削除の選択を変更しても、最後の確定操作まで公開状態を変更しない。確定の処理中は、同じ変更を重ねて実行できない。
- ユーザーIDまたはニーモニックが長い場合でも、両方を識別でき、操作ボタンや画面幅を損なわない。
- 幅320px以上の画面で、公開一覧、申請導線、会話作成者向け操作が横スクロールなく利用できる。
- 会話データまたは申請情報の取得が遅延・失敗した場合、閲覧可能な既知の情報を保ちつつ、再試行できる日本語の案内を表示する。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、会話ごとに公開の「モデレーター」画面を提供しなければならない。
- **FR-002**: モデレーター画面は、ログイン状態および権限にかかわらず、「モデレーターをしているユーザー」と「申請中のユーザー」の二つの一覧を表示しなければならない。
- **FR-003**: 各ユーザー行は、ニーモニックと完全なユーザーIDを同じ視覚的なまとまりで表示しなければならない。ユーザーIDは必要に応じて省略表示してよいが、完全な値を確認およびコピーできなければならない。
- **FR-004**: 申請中のユーザー行は、申請日時および申請理由を公開表示しなければならない。
- **FR-005**: システムは、同一ユーザーをモデレーター一覧と申請中一覧の両方に同時表示してはならない。モデレーターをしている状態を優先して分類しなければならない。
- **FR-006**: ログイン済みで会話作成者でもモデレーターでもなく、申請中でもない一般ユーザーにだけ、モデレーター申請の入力欄と送信操作を提示しなければならない。
- **FR-007**: 未ログインユーザーには公開一覧を表示し、申請操作の位置に、申請のための明確で一つのログイン導線を提示しなければならない。
- **FR-008**: 申請済みの一般ユーザーには、重複送信操作ではなく、自身が申請中であることを表示しなければならない。
- **FR-009**: システムは、現在の34550イベントの発行時刻以後に発行された申請だけを申請中一覧に表示しなければならない。同じ発行時刻の申請は表示対象とする。
- **FR-010**: 会話作成者にだけ、申請中ユーザーの許可対象選択、ユーザーIDによるモデレーター直接追加、既存モデレーターの削除対象選択を提示しなければならない。
- **FR-011**: 許可対象と削除対象はチェックボックスで選択できなければならない。各行に許可または削除を即時実行するボタンを表示してはならない。
- **FR-012**: 会話作成者は、許可、直接追加、削除の保留中の変更を確認し、最後に一つの「変更を確定」操作でまとめて反映できなければならない。変更を確定するまで公開のモデレーター状態を変更してはならない。
- **FR-013**: 変更の確定は、許可対象を申請中一覧からモデレーター一覧へ移動させ、削除対象をモデレーター一覧から除かなければならない。許可済みの申請に再度の許可操作を表示してはならない。
- **FR-014**: 会話作成者以外の利用者には、許可・直接追加・削除の選択操作および変更確定操作を表示してはならない。
- **FR-015**: システムは、モデレーター申請に「却下」操作、却下状態、または `unapproved` のような否定的な内部状態表示を提供してはならない。
- **FR-016**: モデレーターの削除は、明確に「削除」と表示し、会話作成者が削除対象を選択したうえで変更を確定して実行しなければならない。削除操作を申請の却下として表現してはならない。
- **FR-017**: 基本情報の管理画面は会話作成者にだけ表示し、タイトル・説明の編集、掲載申請、会話削除を提供しなければならない。モデレーターの一覧・申請・管理操作は含めてはならない。
- **FR-018**: 会話作成者以外の利用者には、基本情報の管理タブおよび管理フォームを表示してはならない。管理URLへの直接アクセス時には編集可能な情報を表示せず、会話ページへ戻る導線を提供しなければならない。
- **FR-019**: 会話関連のナビゲーションは、全利用者に会話、監査ログ、モデレーターを提示し、会話作成者にだけ基本情報の管理入口を追加しなければならない。
- **FR-020**: ユーザー向けの権限主体の呼称は「会話作成者」に統一し、「管理者」を同じ意味で使用してはならない。
- **FR-021**: モデレーターの追加・許可・削除は、一つの一貫した現在のモデレーター状態を更新しなければならない。会話の基本情報を保存しても、直近のモデレーター変更を失ってはならない。
- **FR-022**: ページは、読み込み中、取得遅延、取得失敗、空の一覧を区別し、利用者が次に取る行動を理解できる日本語の案内と、必要な再試行操作を提示しなければならない。
- **FR-023**: すべての操作コントロールはアクセシブルな名前、キーボードでの操作、視認可能なフォーカス表示を備え、実用上可能な場合は44px以上のタッチ領域を備えなければならない。
- **FR-024**: ルビを除くユーザー向けテキストは14px以上とし、モバイルを含む対象画面幅で主要コンテンツと操作を横スクロールなしに利用できなければならない。

### Key Entities

- **会話作成者**: 会話を作成した利用者。会話の基本情報とモデレーター構成を変更できる唯一の主体。
- **モデレーター**: 現在その会話でモデレーター権限を持つ公開ユーザー。ニーモニックとユーザーIDで表示される。
- **モデレーター申請**: 一般ユーザーがモデレーターへの就任を求める公開の申請。申請者、申請日時、申請理由を持ち、現在の34550イベントの発行時刻以後に発行されたものだけが申請中として表示される。
- **保留中のモデレーター変更**: 会話作成者がチェックボックスまたは直接追加で選択した、許可・追加・削除の未確定の変更内容。最後の確定操作まで公開状態に反映されない。
- **基本情報管理**: 会話のタイトル、説明、掲載申請、会話削除を扱う、会話作成者専用の操作領域。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 未ログイン、ログイン済み一般ユーザー、会話作成者の各状態で、モデレーター画面の公開一覧を100%閲覧できる。
- **SC-002**: 定義済みのモデレーター状態（空、申請のみ、モデレーターのみ、両方あり、変更確定直後）すべてで、各利用者が二つの一覧の意味を正しく識別できる割合を100%にする。
- **SC-003**: 定義済みの許可・直接追加・削除の確定後、対象ユーザーが二つの一覧に重複または未分類で表示されるケースを0件にする。
- **SC-004**: ログイン済み一般ユーザーがモデレーター申請を開始してから、自分の申請中状態を確認するまでを3操作以内で完了できる。
- **SC-005**: 会話作成者が申請の許可、直接追加、削除を選択し、一つの確定操作までを各4操作以内で開始できる。
- **SC-006**: 会話作成者以外の利用者に基本情報管理の編集フォームまたは危険な操作が表示されるケースを0件にする。
- **SC-007**: 幅320px、390px、1440pxの各検証環境で、主要コンテンツの横方向オーバーフローを0件にする。
- **SC-008**: キーボード操作と支援技術による確認で、主要な一覧、状態、申請、会話作成者向け操作に重大なアクセシビリティ阻害が0件であることを確認する。

## Assumptions

- 会話の基本情報自体は既存の会話ページで全利用者に公開済みであり、今回非作成者に見せない対象はその編集・掲載申請・削除の管理UIである。
- 「すべての申請・承認済みは公開」は、現在有効な申請および現在モデレーターをしているユーザーの情報と申請理由を公開する意味として扱う。過去の取消・削除履歴を独立した一覧に追加することは本機能の対象外とする。
- 申請中の表示は、現在の34550イベントの発行時刻を基準に導出する。その時刻より前の申請は表示せず、同時刻の申請は表示する。
- 承認済みの申請者を後からモデレーターから削除しても、削除を反映する34550イベントより前の申請は申請中に戻らない。再度就任を希望する場合は新しい申請を行う。
- 会話作成者は会話ごとの唯一の管理主体であり、グローバルな「管理者」ロールを本機能に導入しない。
- 認証、公開情報の取得、会話情報の保存・掲載申請・削除の既存の仕組みは維持し、本機能では画面構成、状態分類、権限提示を再設計する。
- 画面文言は日本語を正本とし、既存のルビ表示機能と互換性を保つ。
