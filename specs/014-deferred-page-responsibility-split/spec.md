# Feature Specification: 後続ページ責務分離

**Feature Branch**: `014-deferred-page-responsibility-split`

**Created**: 2026-07-14

**Status**: Draft

**Input**: User description: "$speckit-specify KISS原則に準拠するリファクタリングを行っています。013仕様で明示的に「後で実施」とされている（ルートページなどの）ものがあるので、これについて取り組んでください。"

## Clarifications

### Session 2026-07-14

- Q: 3領域をどの単位で段階導入するか → A: 経路検索ページ、場所一覧ページ、会話タブの順に各領域を独立して完了する。
- Q: 責務分離後に維持する外部挙動の範囲はどこまでか → A: URL、ディープリンク、ブラウザ履歴、既存の画面操作と表示をすべて維持する。
- Q: 内部分割に伴う見た目の変更をどこまで許容するか → A: 見た目は原則維持し、アクセシビリティ修正や明らかな不具合修正のみ許容する。
- Q: 責務分離による性能退行をどの基準で判定するか → A: `/api/transit` と `/api/geocode` のAPI応答 p95を200ms以内に維持し、実装後のp95がベースラインを10%超上回らない。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 経路検索ページを安全に分割する (Priority: P1)

利用者は、出発地・目的地・日時を指定して経路を検索し、検索結果の再検索・リセット・PDF出力・エラーからの復帰を、ページ内部の責務を分離した後も同じように行える。

**Why this priority**: 経路検索はアプリの主要な利用価値であり、責務分離による保守性向上が利用者の回帰を招いてはならない。

**Independent Test**: 検索前、検索中、成功、経路なし、通信失敗、レート制限、再検索、リセット、PDF出力の各シナリオを画面テストで実行し、利用者の操作と表示状態が維持されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 利用者が出発地・目的地・日時を入力している、**When** 経路検索を実行する、**Then** 検索中表示が示され、成功時には既存と同じ経路情報が表示される。
2. **Given** 経路が存在しない、通信に失敗する、または利用制限に達する、**When** 利用者が画面を確認する、**Then** 原因に応じた日本語の状態表示と、可能な場合は再試行手段が示される。
3. **Given** 経路結果が表示されている、**When** 利用者が再検索、リセット、またはPDF出力を行う、**Then** 操作対象以外の入力・結果状態が意図せず失われず、PDFの意味情報が画面と一致する。

---

### User Story 2 - 場所一覧ページを安全に分割する (Priority: P1)

利用者は、カテゴリ選択、場所の閲覧、現在地または住所による距離順表示、場所詳細の確認を、ページ内部の責務を分離した後も同じように行える。

**Why this priority**: 場所一覧は複数の取得・分類・位置計算・モーダル表示を含むため、責務を分けることで変更影響を限定しやすくなる。

**Independent Test**: 初期読み込み、カテゴリ切替、場所なし、現在地取得成功・拒否・失敗、住所検索成功・空結果・失敗、詳細表示・閉じる操作を画面テストで確認できる。

**Acceptance Scenarios**:

1. **Given** 場所一覧を開いている、**When** 利用者がカテゴリを切り替える、**Then** 選択したカテゴリの場所だけが表示され、選択状態がアクセシブルに伝わる。
2. **Given** 利用者が距離順表示を選択する、**When** 現在地または住所の取得が成功する、**Then** 距離の近い順と距離情報が既存の意味で表示される。
3. **Given** 位置情報を取得できない、検索結果が空、または詳細情報の取得に失敗する、**When** 利用者が操作を続ける、**Then** エラーや空状態が表示され、一覧全体が操作不能にならない。

---

### User Story 3 - 会話タブの取得境界を分離する (Priority: P1)

利用者は、会話・監査・管理画面のタブ移動と会話情報の段階的表示を、Nostrからの取得処理を表示から分離した後も利用できる。

**Why this priority**: 会話情報の取得には部分取得、既知データ、完了理由、再読み込みなどの保護すべき複雑性があり、表示整理と通信仕様を混ぜない境界が必要である。

**Independent Test**: 初回表示、既知データ表示、取得中、部分取得、完了、取得失敗、再読み込み、タブのキーボード操作を既存の契約テストと画面テストで確認できる。

**Acceptance Scenarios**:

1. **Given** 会話情報を取得中または部分的に取得している、**When** 利用者が画面を開く、**Then** タブと戻る操作は利用可能なまま、取得状態の意味が明示される。
2. **Given** 一部のrelayが遅延または応答しない、**When** 会話情報の読み込みが完了する、**Then** 未観測の情報を確定的な不在として扱わず、完了理由と再試行手段が維持される。
3. **Given** 利用者がタブをキーボードで操作する、**When** 矢印、Home、End、Enterなどを使用する、**Then** フォーカス、選択状態、遷移先が一貫して示される。

---

### User Story 4 - 分割後の共通UI境界を保つ (Priority: P2)

開発者は、分割したページや会話レイアウトで、共通UIを使う箇所とページ固有の表示を明確に判断できる。利用者は、分割前と同じボタン、入力、カード、モーダルの名前・状態・フォーカス表示を得られる。

**Why this priority**: 画面全体の一括移行は変更範囲を過大にするため、責務分離と同時に必要な境界だけを整理し、KISSの目的を守る。

**Independent Test**: 分割対象の代表画面で、キーボード操作、ARIA状態、loading、エラー、破壊的操作の確認・取消を検証し、直接記述が残る箇所には理由があることを確認できる。

**Acceptance Scenarios**:

1. **Given** 分割対象の操作部品を表示する、**When** 利用者が操作する、**Then** 名前、状態、フォーカス表示、タッチ可能な大きさが既存方針を満たす。
2. **Given** 共通UIでは表現しにくいページ固有の表示がある、**When** 開発者が直接UIを記述する、**Then** その範囲と理由が文書化され、暗黙の状態判定を追加しない。

### Edge Cases

- 検索中に利用者がリセットまたは別の検索を実行しても、古い応答が新しい結果を上書きしない。
- 検索結果が空、時刻不明、乗換あり、徒歩区間ありの場合でも、経路表示とPDF出力の意味が一致する。
- 場所データの取得中にカテゴリ変更や詳細表示を行っても、古い取得結果が現在の選択を上書きしない。
- 現在地の利用拒否、タイムアウト、未対応ブラウザ、住所検索のレート制限を、一覧の表示崩れなしに扱う。
- 会話の取得が再読み込み中に重複して開始されても、古い取得結果・エラー・完了理由が新しい取得状態へ混入しない。
- 既知データを表示した後に最新read結果で内容が訂正されても、部分取得を完全取得と誤認しない。
- 画面幅が狭い場合でも、入力、タブ、一覧、モーダル、エラー操作が横方向に欠落しない。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 経路検索ページは、入力、検索操作、経路取得、結果表示、PDF出力、エラー表示の責務を変更理由が異なる単位で分離し、ページ入口にはURL初期化とイベント接続以外の外部取得・API応答変換・経路計算を残してはならない。
- **FR-002**: 場所一覧ページは、場所データ取得、カテゴリ分類、距離計算・並べ替え、一覧表示、詳細表示、エラー表示の責務を分離し、ページ入口には外部取得・GeoJSON分類・距離計算・地域名計算を残してはならない。
- **FR-003**: 会話タブの表示と会話情報の取得・キャッシュ・完了判定を分離し、タブ表示側にはrelay取得・cache更新・completion判定を残してはならない。
- **FR-004**: 分離後も、経路検索、場所一覧、会話タブの公開されている入力、遷移、表示、エラー、再試行の振る舞いを変更してはならない。
- **FR-005**: 経路検索では、検索要求の世代または同等の仕組みにより、古い非同期結果が現在の入力や結果を上書きしてはならない。
- **FR-006**: 場所一覧では、取得・分類・距離計算の失敗をページ全体のクラッシュにせず、利用者が可能な操作を継続できなければならない。
- **FR-007**: 会話情報の取得は、既知データ、partial、unknown、completion、relayの重複排除、再読み込みの意味を維持し、未観測を不在や未承認と確定してはならない。
- **FR-008**: 分割した各責務の境界は、入力と出力を明示し、無関係なUI状態を共有してはならない。
- **FR-009**: 共通UIを使用する範囲と、ページ固有の直接記述を許容する範囲を分割対象ごとに明示しなければならない。
- **FR-010**: 分割対象のボタン、入力、タブ、モーダル、エラー表示は、アクセシブルな名前、状態、フォーカス表示、キーボード操作、日本語の案内を維持しなければならない。
- **FR-011**: ルビの表示契約、Nostrのread契約、PDF生成契約、認証契約を、責務分離を理由に変更または置換してはならない。
- **FR-012**: 新しい永続化領域や新しいユーザー向け機能を追加せず、既存のデータ取得と一時状態の範囲で整理しなければならない。
- **FR-013**: 変更前に、経路検索、場所一覧、会話タブの成功・失敗・境界・アクセシビリティのテストを追加または更新しなければならない。
- **FR-014**: 画面全体の直接DaisyUI部品を一括移行せず、今回の分割で変更する箇所に限って共通UI境界を整理しなければならない。
- **FR-015**: Ruby外部ライブラリの契約変更または置換を本機能の実施範囲に含めてはならない。
- **FR-016**: 実装と受け入れ検証は、経路検索ページ、場所一覧ページ、会話タブの順に各領域を独立して完了させ、先行領域の回帰確認後に次の領域へ進まなければならない。
- **FR-017**: 責務分離後も、URL、ディープリンク、ブラウザ履歴、既存の画面操作、表示内容、エラー状態を変更してはならない。
- **FR-018**: 責務分離に伴う見た目の変更は原則として行わず、アクセシビリティ適合または明らかな表示不具合の修正に限って許容しなければならない。

### Key Entities *(include if feature involves data)*

- **経路検索状態**: 入力、検索中、成功、空結果、制限、通信失敗、再試行可能性を表す利用者向け状態。
- **場所一覧状態**: 場所データ、カテゴリ、距離順、位置取得、詳細表示、空・失敗状態を表す状態。
- **会話メタデータ取得状態**: 会話情報、既知データ、取得中、partial、unknown、completion、エラー、再読み込みを表す状態。
- **責務境界**: データ取得、状態遷移、操作、表示の間で受け渡す明示的な入力・出力。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 経路検索、場所一覧、会話タブの既存主要シナリオについて、分割前後で100%の受け入れシナリオが成立する。
- **SC-002**: 分割対象の各ページ入口が外部取得・ドメイン計算・relay/cache操作を直接行わず、データ取得・状態遷移・利用者操作・表示の各境界に独立したテストがあり、1つの責務変更が無関係な領域のテスト変更を要求しない。
- **SC-003**: 経路検索、場所一覧、会話タブの失敗・再試行シナリオで、ページ再読み込みやページ再訪を行わずに復帰できることを対象シナリオの100%で確認する。
- **SC-004**: キーボード操作とアクセシブルな名前・状態を確認する分割対象の自動テストが、対象シナリオの100%で通過する。
- **SC-005**: 分割後に新規の永続化領域を0件、新規のNostrイベント形式を0件、既存契約の意図しない変更を0件とする。
- **SC-006**: 分割対象のコードレビューで、未使用状態、重複した非同期処理、責務をまたぐ暗黙のclassName判定が新規に増えていない。
- **SC-007**: `/api/transit` と `/api/geocode` の代表シナリオについて、実装前後の同一環境測定でAPI応答 p95 を200ms以内に保ち、実装後のp95がベースラインを10%超上回らない。

## Assumptions

- 対象は、013仕様で延期された経路検索ページ、場所一覧ページ、会話タブの取得境界の責務分離である。
- 既存の画面文言、URL、Nostr relay、セッション状態、データベース、PDF生成、認証の外部契約は再利用する。
- 画面全体の共通UI一括移行は本機能に含めず、分割対象を変更する際の境界整理だけを行う。
- Ruby外部ライブラリは現行契約を保護し、別機能で扱う。
- モバイル、タブレット、デスクトップの既存レスポンシブ挙動と、ルビを除く最小フォントサイズ14pxの方針を維持する。
- 既存のAPI応答 p95 200ms以内の性能方針を本機能にも適用する。
- テスト先行で進め、各責務の境界を実装する前に利用者向け振る舞いと保護すべき契約をテストで固定する。
