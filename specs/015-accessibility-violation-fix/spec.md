## User Scenarios & Testing

### User Story 1 - 主要操作を支援技術で利用する (Priority: P1)

キーボード操作や読み上げを利用する人が、設定、場所検索、会話作成・編集の主要操作を、視覚的なアイコンやマウス操作に依存せず完了できる。

**Why this priority**: 憲章はウェブページ全体を WCAG 2.2 AA の対象とし、名前・役割・値、キーボード操作、入力ラベルを必須確認事項としている。主要操作が利用できないことは機能利用を直接妨げる。

**Independent Test**: 支援技術を有効にした状態とキーボードだけの状態で、対象画面の主要操作を順に実行し、各操作の目的・状態・結果が把握できることを確認する。

**Acceptance Scenarios**:

1. **Given** 設定画面を開いている、**When** ログアウトまたはログイン案内をキーボードで操作する、**Then** 操作可能な要素に到達でき、アイコンだけでなく明確な名前と結果が読み上げられる。
2. **Given** 場所検索画面を開いている、**When** 住所を入力して検索する、**Then** 入力欄の目的が可視ラベルとアクセシブルな名前から分かり、検索結果やエラーが利用者に通知される。

### User Story 2 - 小さな操作を確実に実行する (Priority: P1)

会話を作成・編集する人が、追加済みモデレーターを削除する操作を、指やキーボードで誤操作なく実行できる。

**Why this priority**: 現状の削除操作は 16px 相当の小さなボタンで、AGENTS.md が定める実用的な 44px 以上のタッチターゲットに反する。対象ユーザーの操作を妨げるため、主要操作と同じ優先度で修正する。

**Independent Test**: 会話作成画面と編集画面で、マウス、タッチ相当のポインター、キーボードの各方法により、対象ユーザーを特定して削除できることを確認する。

**Acceptance Scenarios**:

1. **Given** モデレーターが追加済み、**When** そのユーザーの削除操作を実行する、**Then** 操作対象のユーザーを識別できる名前が提示され、44px 以上の操作領域で削除できる。
2. **Given** 削除操作にフォーカスしている、**When** キーボードで実行またはキャンセルする、**Then** フォーカスが可視で、意図しない別ユーザーを削除しない。

### User Story 3 - 一貫した視覚表現と代替情報を得る (Priority: P2)

利用者が、設定、会話作成、場所検索に表示されるアイコンや状態表示を、視覚・非視覚のいずれの方法でも同じ意味として理解できる。

**Why this priority**: 憲章は手書きのインライン SVG を避け、UI の意味を支援技術にも伝えることを求めている。アイコンが情報を伝える場合も、装飾の場合も扱いを統一する必要がある。

**Independent Test**: 対象画面のアイコンを、視覚表示と読み上げの両方で確認し、アイコン単独で意味を伝える箇所と、隣接テキストを補助するだけの箇所を判定する。

**Acceptance Scenarios**:

1. **Given** 状態メッセージにアイコンが表示される、**When** 画面を読み上げる、**Then** 状態の意味がアイコンの読み上げ有無にかかわらずテキストで伝わる。
2. **Given** アイコン付き操作を表示している、**When** キーボードでフォーカスする、**Then** フォーカス表示と操作名が一貫して確認できる。

### User Story 4 - 状態とページ構造を正しく把握する (Priority: P1)

読み上げやキーボード操作を利用する人が、ログインモード、カテゴリ、承認状態、ページ見出し、エラー、成功状態の変化を、視覚的な見た目だけに頼らず把握できる。

**Why this priority**: 再監査で、状態属性の誤記、タブ関連付けの不足、低コントラスト候補、共通ナビゲーションの操作方式の不整合が見つかった。これらは個別画面の装飾ではなく、情報構造と操作結果の理解に影響する。

**Independent Test**: 各対象画面で、ページタイトル、見出し、ランドマーク、タブ状態、フォームエラー、成功・読み込み状態をキーボードと読み上げで確認する。

**Acceptance Scenarios**:

1. **Given** ログインモーダルまたはカテゴリ・承認画面を開いている、**When** 表示中のタブを切り替える、**Then** 選択状態と表示パネルの関係が支援技術に通知される。
2. **Given** 検索または保存でエラー・成功状態が発生する、**When** 画面のフォーカスを移動させずに状態が更新される、**Then** 状態メッセージが重複なく読み上げられる。
3. **Given** 共通レイアウトを表示している、**When** キーボードでメニューを開閉し、本文へ移動する、**Then** 操作可能な順序、フォーカス表示、スキップ先が一貫している。

### Edge Cases

- アイコンが表示できない場合でも、隣接するテキストによって操作や状態の意味を理解できる。
- モデレーター名が長い場合でも、削除ボタンの名前が対象ユーザーを含み、操作領域が縮小しない。
- 入力エラーが発生した場合でも、エラー内容が入力欄と関連付けられ、色だけに依存せず伝わる。
- 操作中に読み込み状態へ変化した場合でも、完了・失敗がステータスメッセージとして伝わる。
- ログインモードやタブを切り替えた場合でも、選択状態が見た目のクラスだけに依存しない。
- 画面幅を狭め、文字サイズを拡大しても、主要情報・操作・エラーが横方向に欠落しない。
- 背景と通常テキスト、アイコン、境界線、フォーカス表示のコントラストが対象テーマで基準を下回らない。

## Requirements

### Functional Requirements

- **FR-001**: 対象範囲のすべての操作可能なコントロールは、視覚表示に依存しない一意で目的の分かるアクセシブルな名前を持たなければならない。
- **FR-001a**: すべての状態属性、役割、値、タブとパネルの関連付けは、使用する標準仕様に従った属性名で実装され、視覚的なクラスだけで状態を表してはならない。
- **FR-002**: 住所検索入力は、可視ラベルと関連付けられ、プレースホルダーだけを入力名や説明の代わりにしてはならない。
- **FR-002a**: グループ化された入力は、そのグループの説明を見出しまたは凡例としてプログラムから判定できるようにしなければならない。
- **FR-003**: モデレーター削除操作は、対象ユーザーを名前に含む操作説明を持ち、操作領域を縦横 44px 以上にしなければならない。
- **FR-004**: 対象範囲の装飾用アイコンは支援技術に重複して読み上げられないようにし、意味を伝える情報はテキストによる代替を持たなければならない。
- **FR-005**: 対象範囲のアイコン表示は、憲章で禁止されている手書きのインライン SVG に依存してはならない。
- **FR-006**: 対象範囲のすべての操作はキーボードで実行でき、フォーカス位置を視認でき、キーボードトラップを発生させてはならない。
- **FR-007**: 入力エラー、読み込み、成功、失敗などの動的な状態は、色だけに依存せず、支援技術に適切なステータスまたはエラーとして通知されなければならない。
- **FR-008**: 通常テキスト、非テキストのUI要素、フォーカス表示のコントラストは WCAG 2.2 AA の該当基準を満たし、色だけで情報やエラーを伝えてはならない。
- **FR-009**: 対象範囲のページはページタイトル、ページ言語、見出し階層、スキップ手段、ランドマーク、フォーカス順序を一貫して提供しなければならない。
- **FR-010**: 文字サイズを 200% まで拡大し、狭い画面幅で表示しても、主要操作・フォーム・エラー・状態メッセージが利用できなければならない。
- **FR-011**: 既存の会話作成、会話編集、設定、場所検索の利用目的とデータ処理を変更せず、アクセシビリティ上の修正に範囲を限定しなければならない。

### Key Entities

- **操作コントロール**: 利用者がクリック、タップ、またはキーボードで実行するボタン、リンク、入力欄、タブ。
- **状態メッセージ**: 検索、読み込み、入力エラー、成功、失敗など、操作の結果や現在状態を示す情報。
- **モデレーター対象者**: 会話作成・編集画面で追加済みとして表示され、個別に削除されるユーザー。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 対象画面で、キーボードだけを使う受け入れテストの 100% が、対象とする主要操作を完了できる。
- **SC-002**: 対象範囲の操作可能なコントロールの 100% が、目的を識別できるアクセシブルな名前、可視フォーカス、キーボード操作を満たす。
- **SC-003**: モデレーター削除操作の 100% が 44px 以上の操作領域と対象ユーザーを含む操作名を持つ。
- **SC-004**: 対象範囲で検出される手書きインライン SVG、ラベル未関連付け入力、アイコンだけの名前なし操作が 0 件になる。
- **SC-005**: 支援技術を使った受け入れテストで、入力エラーと主要な成功・失敗状態を 100% テキストとして識別できる。
- **SC-006**: 対象範囲で、誤った ARIA 属性名、タブとパネルの関連付け不足、キーボードで到達できない操作が 0 件になる。
- **SC-007**: WCAG 2.2 AA の対象となる通常テキスト、非テキストUI、フォーカス表示のコントラスト検査で、基準未達の組み合わせが 0 件になる。
- **SC-008**: 200% の文字サイズと狭い画面幅の受け入れテストで、横スクロールによる情報・操作の欠落が 0 件になる。

## Assumptions

- 対象範囲は、現時点で違反を確認した設定、場所検索、会話作成、会話編集の画面に加え、全ページで共有されるレイアウト、ナビゲーション、モーダル、入力、タブ、ボタン、状態表示とする。
- 目標は憲章に定める WCAG 2.2 AA であり、対象画面に関係する達成基準の本文を確認して受け入れ判定する。
- 既存の認証、会話、場所データ、画面遷移、エラー処理の機能は再利用し、永続データの形式や保持方針は変更しない。
- アイコンの視認性は維持するが、アイコン自体を情報の唯一の伝達手段にはしない。

## Out of Scope

- アクセシビリティ修正を目的としないページの大規模なデザイン刷新。
- 新しい認証方式、保存先、データモデル、業務フローの追加。
- WCAG 2.2 AA の対象外となる新機能の追加。
