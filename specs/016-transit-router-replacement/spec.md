# Feature Specification: 公共交通経路探索エンジンの置換

**Feature Branch**: `dev`

**Created**: 2026-07-16

**Status**: Draft

**Input**: User description: "$speckit-specify issue #29 の独自実装された案内アルゴリズムを、弱いサーバーでも扱える簡素な既存ライブラリへ置換する。既存機能への影響を確認し、徒歩の道路経路探索や新たな交通機関対応は追加しない。"

**Related Issue**: [#29 案内アルゴリズムが独自実装でメンテナンスしづらい](https://github.com/nawashiro/kazaguruma-transit/issues/29)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - これまでどおり公共交通経路を検索する (Priority: P1)

利用者は、経路探索の内部方式が置き換わった後も、出発地、目的地、日時、出発時刻指定または到着時刻指定を使って、現在提供されている公共交通経路を検索できる。結果には利用する便、乗車・降車停留所、出発・到着時刻、乗換、所要時間が従来と同じ意味で示される。

**Why this priority**: 本機能は保守性改善を目的とする内部変更であり、既存の経路検索を失わないことが最優先である。

**Independent Test**: 固定した時刻表データに対して、直通、乗換、経路なし、運行日例外、出発時刻指定、到着時刻指定の代表検索を実行し、期待する旅程と時刻が返ることを単独で確認できる。

**Acceptance Scenarios**:

1. **Given** 指定日に運行する直通便がある、**When** 利用者が出発時刻を指定して検索する、**Then** 指定時刻以降に利用可能な直通経路と正しい乗降時刻が表示される。
2. **Given** 直通便がなく、許容範囲内の乗換で到達できる、**When** 利用者が経路を検索する、**Then** 乗換停留所、各便、待ち時間、最終到着時刻を含む経路が表示される。
3. **Given** 指定時刻までに目的地へ到着できる便がある、**When** 利用者が到着時刻を指定して検索する、**Then** 到着条件を満たす経路が表示される。
4. **Given** 指定日が通常運行日または運行例外日である、**When** 利用者が経路を検索する、**Then** その日に有効な便だけが候補になる。
5. **Given** 指定条件で利用可能な経路がない、**When** 検索が完了する、**Then** 存在しない旅程を生成せず、経路が見つからない状態が示される。

---

### User Story 2 - 現在の検索方針と表示契約を維持する (Priority: P1)

利用者は、最寄り停留所を使う通常検索、周辺停留所を含む速さ優先検索、直通優先、乗換回数の上限、検索時間範囲といった現在の検索方針を引き続き利用できる。画面、PDF、関連機能へ渡される経路情報の意味も変わらない。

**Why this priority**: 経路が一件返るだけでは互換とはいえず、候補選択と既存の出力契約が変わると利用者の判断や下流機能に回帰が生じるためである。

**Independent Test**: 同じ入力に複数候補を含む時刻表を用意し、通常検索と速さ優先検索の選択結果、乗換上限、徒歩アクセス見積り、画面・PDF用情報の整合性を確認できる。

**Acceptance Scenarios**:

1. **Given** 直通経路と乗換経路の両方が利用可能である、**When** 利用者が通常検索を行う、**Then** 現在の方針どおり直通経路が優先される。
2. **Given** 周辺の複数停留所から異なる経路が利用可能である、**When** 利用者が速さ優先で検索する、**Then** 出発地・目的地と停留所間の移動見積りを含む総所要時間に基づいて候補が選ばれる。
3. **Given** 乗換回数または検索時間範囲を超える経路しかない、**When** 検索する、**Then** その経路は結果に含まれない。
4. **Given** 経路検索が成功した、**When** 同じ経路を画面表示またはPDF出力に利用する、**Then** 便、停留所、時刻、乗換、徒歩アクセス区間の意味が一致する。

---

### User Story 3 - 小規模な運用環境で安定して検索する (Priority: P2)

運用者は、現在の小規模なサーバー構成を大幅に増強したり、利用者の検索ごとに外部の経路検索サービスへ問い合わせたりせずに、日々の時刻表を使った経路案内を提供できる。

**Why this priority**: 高機能な複合交通ルーターを導入して運用負荷を増やすことは、プロジェクトの規模とサーバー制約に合わないためである。

**Independent Test**: 実運用相当の一日分の時刻表を読み込み、現在と同じ資源上限の検証環境で連続検索を実行し、応答性能、メモリ上限、外部経路検索への依存がないことを確認できる。

**Acceptance Scenarios**:

1. **Given** 実運用相当の一日分の時刻表が利用可能である、**When** サービスを起動して代表経路を連続検索する、**Then** 現在の運用資源上限内で処理が継続し、資源不足で終了しない。
2. **Given** 時刻表データが既に取り込まれている、**When** 利用者が経路を検索する、**Then** 検索ごとに外部の経路探索サービスへ位置情報や検索条件を送信しない。
3. **Given** 対象外の交通モードを含む経路が考えられる、**When** 利用者が検索する、**Then** 現在提供していない交通モードを自動的に追加した複合経路を返さない。

---

### User Story 4 - 独自アルゴリズムを保守せずに変更できる (Priority: P2)

保守担当者は、時刻表に基づくグラフ探索そのものをプロジェクト内で改修せず、経路検索の入力、結果変換、既存の選択方針というプロジェクト固有部分に集中できる。

**Why this priority**: issue #29の主目的は、独自アルゴリズムの理解・修正・検証を継続的に負担する状態を解消することだからである。

**Independent Test**: プロジェクト内の経路探索責務を確認し、独自の最短経路探索が残っておらず、プロジェクト固有の入出力と選択方針を一つの境界で検証できることを確認する。

**Acceptance Scenarios**:

1. **Given** 経路探索部品の入力または出力形式が変更される、**When** 保守担当者が追随する、**Then** 変更箇所を経路探索の統合境界に限定できる。
2. **Given** 経路結果の表示項目を変更する、**When** 保守担当者が変換規則を更新する、**Then** 最短経路探索の内部処理を変更せず対応できる。
3. **Given** 経路探索部品を更新する、**When** 回帰検証を行う、**Then** 固定した代表時刻表と実運用相当データの両方で既存の検索契約を確認できる。

### Edge Cases

- 出発停留所と目的停留所が同一の場合、不要な乗車経路を生成しない。
- 指定時刻が日付をまたぐ便、24時を超える時刻表表現を含む場合でも、運行日と時刻を誤って翌日または前日へずらさない。
- 通常運行を追加・削除する運行日例外がある場合、その例外を優先する。
- 同名停留所、親停留所と乗降場、往路・復路で異なる停留所IDがある場合、誤った停留所を接続しない。
- 乗換時間が不足する接続や、乗換上限を超える候補を採用しない。
- 一部の便に到着時刻または出発時刻が欠ける場合、推測した時刻を確定値として表示しない。
- 時刻表が空、不完全、読み込み不能の場合、サービス全体を不安定にせず、検索不能であることを記録して利用者へ理解可能な失敗結果を返す。
- 周辺停留所が見つからない場合、無制限に探索範囲を拡大しない。
- 連続する同一便や車内継続が存在する場合、不要な乗換として数えない。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、設定済みの時刻表データに基づいて、出発停留所から目的停留所までの利用可能な公共交通経路を検索できなければならない。
- **FR-002**: システムは、出発時刻指定と到着時刻指定の両方を維持し、それぞれの指定時刻を満たす経路だけを返さなければならない。
- **FR-003**: システムは、通常運行カレンダーと運行日例外を反映し、指定日に運行しない便を経路へ含めてはならない。
- **FR-004**: システムは、直通経路と乗換経路を扱い、現在の上限である最大2回の乗換と180分の検索時間範囲を維持しなければならない。
- **FR-005**: 通常検索では、利用可能な直通経路がある場合に直通を優先し、直通がない場合に乗換経路から指定条件に最適な候補を選ばなければならない。
- **FR-006**: 速さ優先検索では、出発地・目的地周辺の既定範囲内にある停留所候補を比較し、停留所までと停留所からの移動見積りを含む総所要時間に基づいて候補を選ばなければならない。
- **FR-007**: 経路結果は、便、路線、乗車・降車・乗換停留所、各区間の出発・到着時刻、乗換回数、総所要時間を、既存の利用箇所が同じ意味で扱える形で提供しなければならない。
- **FR-008**: 経路が存在しない場合、または時刻表を利用できない場合、システムは架空の経路を生成せず、経路なしと処理失敗を区別できる結果を提供しなければならない。
- **FR-009**: 経路検索は、利用者ごとの検索条件や位置情報を外部の経路探索サービスへ送信せずに完了しなければならない。
- **FR-010**: 経路探索の置換後、プロジェクト内に独自の最短経路探索または時刻表グラフ探索を正規の検索経路として残してはならない。
- **FR-011**: プロジェクト固有の入力変換、結果変換、候補選択、エラー処理は、一貫した経路検索境界を通して検証可能でなければならない。
- **FR-012**: 経路探索部品の更新は意図しない版へ自動的に拡大せず、対応する実行環境と回帰確認済みの版を再現可能でなければならない。
- **FR-013**: 置換前に、直通、乗換、経路なし、運行日例外、出発時刻指定、到着時刻指定、日付またぎ、周辺停留所検索の期待結果を回帰データとして固定しなければならない。
- **FR-014**: 実運用相当の時刻表を使った回帰確認を、経路探索部品の導入時および更新時に再実行できなければならない。
- **FR-015**: 既存の画面表示、PDF出力、時刻表表示、停留所情報、認証、会話、評価、レート制限の契約を本機能によって変更してはならない。
- **FR-016**: 本機能は、停留所まで・停留所からの既存の距離および時間見積りを維持する一方、道路網に基づく徒歩経路探索を新たに提供してはならない。
- **FR-017**: 本機能は、現在の設定済み時刻表が対象としない交通機関や、複数交通モードを組み合わせる新しい案内を追加してはならない。
- **FR-018**: 時刻表の読み込みまたは経路検索で回復可能な失敗が起きた場合、他の利用者向け機能を巻き込んで停止せず、原因を運用者が識別できなければならない。

### Key Entities

- **経路検索条件**: 出発地、目的地、検索日時、出発・到着指定、速さ優先の有無を表す。
- **運行日スナップショット**: 指定日の通常運行と運行日例外を反映した、検索対象となる便・停留所・時刻の集合。
- **停留所候補**: 出発地または目的地の近隣で、経路探索の始点・終点になり得る停留所と移動見積り。
- **旅程**: 一つ以上の乗車区間、必要な乗換、出発・到着時刻、総所要時間からなる検索結果。
- **乗車区間**: 同一便で移動する乗車停留所から降車停留所までの区間と、その路線・便・時刻情報。
- **経路なし結果**: 正常に検索したが条件を満たす旅程が存在しない状態。時刻表読み込み失敗などの処理失敗とは区別する。
- **回帰データセット**: 置換前後で検索結果の意味を比較するための、固定時刻表、検索条件、期待結果の組。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 直通、1回乗換、2回乗換、経路なし、通常運行日、運行例外日、出発時刻指定、到着時刻指定、日付またぎの代表シナリオを各1件以上含む回帰データセットで、期待する到達可否、便、停留所、時刻、乗換回数が100%一致する。
- **SC-002**: 通常検索、速さ優先検索、周辺停留所フォールバックの代表シナリオで、候補選択方針と総所要時間の意味が100%維持される。
- **SC-003**: 実運用相当の一日分の時刻表を使用した経路検索の95%以上が200ミリ秒以内に完了する。
- **SC-004**: 実運用相当データの読み込み後に代表検索を100回連続実行しても、現在の本番資源上限を超えず、資源不足による終了が0件である。ピークメモリが置換前の同条件ベースラインから20%を超えて増加する場合は不合格とする。
- **SC-005**: 経路検索時に、外部の経路探索サービスへ送信される検索条件および位置情報が0件である。
- **SC-006**: 既存の画面表示、PDF出力、時刻表、停留所情報、認証、会話、評価、レート制限に関する自動回帰検証の失敗が0件である。
- **SC-007**: プロジェクト内で正規の検索に利用される独自の最短経路探索実装が0件となり、経路探索部品との統合境界が1箇所に限定される。
- **SC-008**: 徒歩の道路経路探索または新しい交通モードに関する利用者向け機能追加が0件である。
- **SC-009**: 時刻表読み込み不能、空データ、不正な停留所、経路なしの各代表ケースで、サービス全体の異常終了が0件であり、経路なしと処理失敗を100%区別できる。

## Assumptions

- issue #29に記載されたリポジトリ公開は完了済みであり、本仕様は独自経路探索の置換を対象とする。
- 現在取り込んでいるGTFS時刻表と既存の停留所・路線識別子を引き続き正本として使用し、新しい永続データは追加しない。
- 本機能ではサーバー実行環境のメジャー更新を行わず、既存の認証依存を含めて正式対応する現在の実行環境を維持する。
- 採用候補は、現在の実行環境と実GTFSによる事前検証で、時刻表解析と代表経路探索に成功している。
- 経路探索部品が直接提供しない既存の検索方針は、統合境界で意味を維持する。
- 停留所まで・停留所からの徒歩距離と時間は、現在の簡易見積りを維持する。道路形状、段差、横断歩道などを考慮した徒歩ナビゲーションは対象外とする。
- 現在のGTFS取込、データベース、利用者向け応答形式、画面とPDFの表示契約を維持する。
- 経路探索部品のライセンス表示と配布条件は、既存のライセンス一覧生成に含められるものとする。
